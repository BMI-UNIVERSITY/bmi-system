# Database Migrations Guide

The BMI System uses Cloudflare D1 as its primary database. This document outlines the authoritative source of truth for our database schema and provides guidelines for creating, applying, and verifying migrations.

## Authoritative Source of Truth
The **`apps/api/migrations/`** directory is the **single source of truth** for the database schema. 
All changes to the database structure must be represented as sequentially numbered SQL files in this directory. 

We do not maintain a separate `schema.sql` dump file because Cloudflare's `wrangler` CLI automatically tracks the applied migrations using its internal `d1_migrations` table.

## Creating a New Migration
When you need to alter the database schema (add a table, add a column, modify an index, etc.):

1. Navigate to the API application:
   ```bash
   cd apps/api
   ```
2. Generate a new migration file using Wrangler:
   ```bash
   npx wrangler d1 migrations create bmi-portal-db <migration_name>
   ```
   *Replace `<migration_name>` with a descriptive name, e.g., `add_status_to_users`.*
3. A new empty `.sql` file will be created in `apps/api/migrations/`. Open it and write your raw SQL statements.

### Writing Idempotent Migrations
Where possible, write migrations that can be run multiple times safely. Use `IF NOT EXISTS` for tables and indexes:
```sql
CREATE TABLE IF NOT EXISTS new_table (...);
CREATE INDEX IF NOT EXISTS idx_new_table ON new_table(id);
```
*(Note: SQLite does not support `IF NOT EXISTS` for `ALTER TABLE ADD COLUMN`. Wrangler's migration tracker handles this by only executing new files, but keep it in mind.)*

## Applying Migrations

### Local Environment
To apply new migrations to your local development database:
```bash
cd apps/api
npx wrangler d1 migrations apply bmi-portal-db --local
```

### Production Environment
To apply migrations to the production D1 database:
```bash
cd apps/api
npx wrangler d1 migrations apply bmi-portal-db --remote
```

## Verifying Migrations in CI/CD
To ensure that all migrations are syntactically valid and apply cleanly before code is merged, we use a verification script.

You can manually verify all migrations from a clean slate by running:
```bash
npm run verify-migrations
```
This script spawns a temporary in-memory database or isolates a local database environment, applies all migrations in order, and validates that no errors occur. It is automatically run during continuous integration.

## Future Tooling (ORMs)
Currently, migrations are written in raw SQL. As the project scales, we may adopt an ORM (such as **Drizzle ORM**). Drizzle provides native support for Cloudflare D1 and can automatically generate migration files from TypeScript schemas. If/when this transition occurs, the `migrations` folder will remain the source of truth, but the SQL files will be autogenerated rather than hand-written.
